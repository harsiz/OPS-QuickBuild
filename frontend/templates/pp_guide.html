What this is

The PP builder lets you redefine how your server awards performance points (pp) without writing a line of code. You stack blocks into a flow; the flow takes the pp osu! would normally give a play and transforms it into the pp your server gives.

Under the hood the builder turns your blocks into a Python function called modify_pp(pp, ctx). It runs in two places, always identically:

  • Live — every time a score is submitted in-game.
  • Recalc — when you hit Save and Update, every existing score is recomputed so old and new plays live under the same rules.
Nothing you do here can break the game server. If a flow ever errors on a score, that score silently falls back to normal bancho pp — it "fails open".

How osu! pp works (the 2-minute version)

You can't tweak pp meaningfully without knowing what the number you're given represents. When a play finishes, osu!'s calculator looks at the beatmap and your score and produces:

  • Star rating (stars) — the map's overall difficulty.
  • Skill components — the raw pp is built from separate skills: pp_aim (jumps/flow), pp_speed (streams/tapping), pp_acc (precision), and pp_flashlight. These roughly add up to the total.
  • The total — combined and scaled by your accuracy, combo and misses. This total is the pp value handed to your flow.

So a 200pp jump map and a 200pp stream map arrive with the same pp but very different pp_aim/pp_speed splits. That split is what lets you buff streams and nerf aim — more on that below.

The execution model

Your flow runs top to bottom, once per score. Two values matter:

valuemeaning
ppthe original bancho pp. Never changes — read it any time.
new_ppthe running result. Every block edits this. What's in it at the end is what the play is worth.

When the flow starts, new_pp equals pp. A block like ✖️ multiply pp with formula 1.5 does new_pp = new_pp × 1.5. Order matters: multiplying by 2 and then capping at 1000 is different from capping first.

At the very end the server guards the result (negative or invalid → 0) and hands it back. You never write the return — the builder does.

The formula language

Every white box is a formula — a little math expression. It supports:

  • numbers: 1.5, 0.35, 4000
  • arithmetic: + - * / ** (** is "to the power of")
  • comparisons: > < >= <= == !=
  • logic: and, or, not
  • variables and functions (below)
  • parentheses to group: (pp + 100) * 2

In a condition (the if box) the formula is read as true/false. In a value box it's read as a number. A comparison like acc > 98 is 1 when true and 0 when false, so you can even use it inside arithmetic: 1 + 0.2 * (acc > 98) adds 20% only on 98%+ plays.

Division is safe: dividing by zero gives 0 instead of crashing. Only whitelisted names work — a typo like accc is caught the moment you press Compile, never at score time.

No mod & the mods bitmask

"Mods" is a single number where each mod is one bit. Adding mods adds their numbers:

modbitflag
NoFail1has_mod(1)
Easy2is_ez
Hidden8is_hd
HardRock16is_hr
DoubleTime/NC64is_dt
Flashlight1024is_fl
Relax128is_relax

So a play with HD + HR has mods = 8 + 16 = 24. A no mod play has mods = 0 — no bits set at all. That's the key:

no mod  →  mods == 0
has DT  →  is_dt   (i.e. mods & 64)
DT + HD →  is_dt and is_hd

The math of "no mod". When mods == 0, every is_* flag is 0, so any if is_dt, if is_hr, etc. block is skipped entirely. A no-mod play is therefore shaped only by your unconditional blocks — the base multiplier, the stream/aim skew, accuracy and star rules. This is exactly why a "reward no mod" idea works by penalising the modded plays rather than boosting nomod:

✖️ multiply pp by 1.5              (everyone)
❓ if mods == 0:
   ✖️ multiply pp by 1.5           (nomod only → effectively 2.25×)
   else:
   ✖️ multiply pp by 0.8           (any mod → 1.2×)

Because osu! already gives DT and HR plays more raw pp (they're harder), servers often use the builder to flatten that — e.g. nerf is_dt — so nomod skill stays competitive on the leaderboard.

Base multiplier & the ramp curve

Most economies start with a base multiplier that grows with how good the play is. The Auraelia default is:

📦 pv  = ramp(pp, 500)          # 0 at 0pp, 1 at 500+ raw pp
✖️ pp *= 1.5 + 2.0 * (pv ** 0.45)

Read it piece by piece:

  • ramp(pp, 500) maps the raw pp onto a 0→1 "play value" pv. A 100pp play gives pv = 0.2; a 500pp+ play gives pv = 1.
  • 1.5 + 2.0 * pv would be a straight line from 1.5× (weak plays) to 3.5× (elite plays).
  • The ** 0.45 bends that line. An exponent below 1 is front-loaded: the multiplier rises fast for weak/mid plays and crawls at the top. Above 1 is back-loaded (top-heavy). Exactly 1 is a straight line.

Worked numbers with exponent 0.45:

raw pppvpv0.45multiplierresult
1000.200.482.46×246
2500.500.732.96×740
500+1.001.003.50×1750+

Notice a mid play (250) is already ~85% of the way to the max multiplier — that "fast then flat" shape is the front-loading. Want the opposite (only the very best plays get rewarded)? Raise the exponent to 2. Want an even spread? Use 1.

Streams vs aim (the aim/speed split)

Every play arrives with pp_aim and pp_speed. From them we derive:

speed_share = pp_speed / (pp_aim + pp_speed)

speed_share is 0 on a pure aim map, 1 on a pure stream map, and around 0.35–0.45 on a balanced map. To buff streams and nerf aim, measure how far a map leans from neutral (0.40) and scale the effect:

❓ if speed_share > 0.40:
   ✖️ multiply pp by 1 + 0.50 * clamp((speed_share - 0.40) / 0.25, 0, 1)
   else:
   ✖️ multiply pp by 1 - 0.15 * clamp((0.40 - speed_share) / 0.25, 0, 1)

The clamp(…, 0, 1) turns "distance from neutral" into a 0→1 strength that maxes out 0.25 past neutral. So:

  • a map at speed_share = 0.65 is (0.65−0.40)/0.25 = 1.0 → the full +50% stream buff.
  • a map at 0.525 is halfway → +25%.
  • a heavy aim map at 0.15 → the full −15% nerf.

Change 0.40 to move the neutral point, 0.25 to change how quickly the effect ramps (smaller = more sensitive), and the 0.50/0.15 to change the maximum buff/nerf.

Accuracy & misses

acc is a 0–100 percentage. A clean way to reward high accuracy is a ramp over the top slice of the range:

✖️ multiply pp by 1 + 0.20 * clamp((acc - 95) / 5, 0, 1)

This is +0% at 95% and below, scaling to +20% at 100%. The (acc − 95) / 5 maps 95→100 onto 0→1, and clamp stops it going negative below 95 or above 20% past 100.

Misses are a raw count. A per-miss penalty with a floor so one bad play isn't zeroed:

❓ if nmiss > 0:
   ✖️ multiply pp by max(1 - 0.03 * nmiss, 0.70)

Each miss removes 3%, but max(…, 0.70) stops the penalty ever exceeding 30%. A 5-miss play keeps 1 − 0.15 = 0.85; a 20-miss play bottoms out at 0.70.

Star rating

stars lets you shape the economy by map difficulty directly. Two common tools:

❓ if stars > 6:
   ✖️ multiply pp by 1 + min(0.10 * (stars - 6), 0.50)   # +10%/star over 6★, capped +50%
❓ if stars < 3:
   ✖️ multiply pp by 0.40                                 # gut the 2★ farm

The min(…, 0.50) caps the high-star bonus so a 12★ outlier doesn't explode. The low-star nerf is the single most effective anti-farm lever: cheap 1–2★ "relax spinner" maps stop being worth grinding.

Caps & softcaps — the math

A hard cap is a flat wall: new_pp = min(new_pp, cap). Everything above the cap becomes exactly the cap, so a 3800pp and a 6000pp play both read 4000. Simple, but it flattens the very top of your leaderboard into a tie.

A softcap keeps some separation by applying diminishing returns past the cap:

if new_pp > cap:
    new_pp = cap + (new_pp - cap) ** exp

With cap = 1200 and exp = 0.8, the "overflow" above 1200 is raised to the 0.8 power — so it still grows, just slower:

pre-cap ppoverflowoverflow0.8final
1200001200
1400200691269
20008002101410
400028005741774

Lower exp = harsher squeeze (0.5 is very flat); closer to 1 = gentler. The best plays still rank above good plays, just not by a runaway margin.

Relax

Relax plays are flagged by is_relax (the RX mod, or a relax game mode). Because relax removes the hardest skill (tapping accuracy), servers usually discount it so it can share the main leaderboard fairly:

❓ if is_relax:
   ✖️ multiply pp by 0.35        # relax keeps 35% (a 65% reduction)

Put this near the end of the flow so the discount applies to the finished value, after your base and skill rules.

From play pp to profile pp

Your flow sets the pp of a single play. A player's profile pp is built from their scores separately, and it's worth knowing so numbers don't surprise you:

  • Each player's best play per map is taken and sorted highest-first.
  • Only the top 100 count, and each is weighted: the #1 play counts fully, #2 counts 95%, #3 counts 0.95² = 90.25%, and so on (0.95^rank).
  • Those weighted values are summed into profile pp.

Two consequences: (1) total pp is much larger than any single play — a stack of good plays adds up. (2) Because it's a weighted top 100, a multiplier that changes every play also multiplies the total by roughly the same factor, but rank order can still shuffle because different players' top-100s react differently to your rules. A full recalc always re-sorts everyone from scratch.

Build a custom preset from scratch

Let's design a "precision" economy: modest base, big accuracy reward, punish misses, keep the top tidy. Build it block by block.

  1. Base. Add 📦 set variable pv = ramp(pp, 600), then ✖️ multiply pp by 1.0 + 0.6 * pv. That's a gentle 1.0→1.6× range — precision servers keep multipliers low so skill, not inflation, decides ranks.
  2. Accuracy reward. Add ✖️ multiply pp by 1 + 0.35 * clamp((acc - 96) / 4, 0, 1) — up to +35% for a 100% play.
  3. Miss punishment. Add ❓ if nmiss > 0, and inside it ✖️ multiply pp by max(1 - 0.05 * nmiss, 0.5).
  4. Anti-farm. Add ❓ if stars < 3 → ✖️ 0.5.
  5. Relax. Add ❓ if is_relax → ✖️ 0.3.
  6. Tidy top. Add a 🪶 softcap past 1200 exp 0.75.

Press Compile to see the generated Python and confirm it's valid, watch the readable preview match your intent, then Save and Update. That's a complete, original ruleset in six blocks.

Tip: start from the closest preset (the search box up top) and adjust, rather than from an empty flow. "Tryhard" is a great base for competitive tweaks.

Compile, save & apply

  • 🔧 Compile — validates your flow and shows the exact Python it generates. Use it freely; it changes nothing on the server. Errors (bad name, empty box) appear here in red.
  • 💾 Save and Update — writes the profile (backing up the previous one as .bak), recalculates every score, rebuilds leaderboards, and restarts the game server. Players are briefly disconnected by the restart and can rejoin immediately.

Saving runs in the background — the last-update log on the PP page shows progress and a "done" line when the new economy is live (a full recalc takes a few minutes).

Gotchas

  • Order is everything. Caps, softcaps and the relax discount belong at the end; the base multiplier at the start. A cap placed early gets undone by later multiplies.
  • Variable names must be lowercase letters/numbers/underscores and can't reuse a built-in name (like ramp or min). Set a variable before you read it.
  • Percent vs multiplier. "+20%" is × 1.2, not × 0.2. A nerf of 30% is × 0.7.
  • Big multipliers + no cap can send top plays into the tens of thousands. That's fine if intended — just know a recalc will reshuffle the whole leaderboard.
  • When in doubt, Compile and read the generated Python. It never lies about what your blocks actually do.
